# Security Audit Skill

## Overview

앱 코드베이스를 분석하여 보안 취약점을 식별하고, 프로젝트 유형에 적합한 보안 하드닝을 적용한다.

## Workflow

### Phase 1: 앱 유형 판별 (자동)

코드베이스를 스캔하여 앱 유형을 판별한다:

| 신호 | 판별 |
|------|------|
| `next.config.*`, `nuxt.config.*`, `vite.config.*` + 서버 API 라우트 존재 | **웹앱 (풀스택)** |
| `express`, `fastify`, `hono` 등 서버 프레임워크만 존재 | **API 서버** |
| `electron`, `main.js` + `BrowserWindow` | **일렉트론 앱** |
| 정적 HTML/JS만 존재 (서버 없음) | **정적 사이트** |

### Phase 2: 현재 보안 상태 감사

코드베이스에서 다음을 스캔한다:

1. **인증/인가 패턴**: auth 미들웨어, 세션 관리, 역할 검사 존재 여부
2. **입력 검증**: Zod/Joi/Yup 스키마 또는 수동 검증 존재 여부
3. **에러 처리**: try-catch, 에러 응답에 내부 정보 노출 여부
4. **보안 헤더**: CSP, HSTS, X-Frame-Options 설정 여부
5. **환경 변수**: 하드코딩된 시크릿, .env 파일 gitignore 여부
6. **의존성**: `npm audit` 결과

### Phase 3: 갭 분석 리포트 생성

`references/checklist.md`를 참조하여, 앱 유형에 해당하는 항목 중 누락된 것을 리포트한다.

리포트 형식:
```
## 보안 감사 결과

### 앱 유형: [웹앱/API/일렉트론/정적]

### 필수 (미적용)
- [ ] 항목명 — 위험도, 설명

### 필수 (적용 완료)
- [x] 항목명

### 권장 (선택)
- [ ] 항목명 — 설명

### 해당 없음
- 항목명 — 이유
```

**중요**: 리포트를 사용자에게 보여주고 적용 여부를 확인받은 후 Phase 4로 진행한다.

### Phase 4: 보안 하드닝 적용

사용자 승인된 항목만 적용한다. 적용 순서:

1. 입력 검증 (Zod 스키마)
2. 인증/인가 가드
3. 에러 처리 래퍼
4. 보안 헤더/미들웨어
5. 쿠키/세션 설정
6. 기타 항목

### Phase 5: 검증

1. `npx next build` (또는 해당 빌드 명령) 실행
2. `npm audit` 실행
3. 결과 리포트

## Notes

- 이 스킬은 **코드를 직접 수정**한다. Phase 3 리포트 후 반드시 사용자 승인을 받는다.
- 프레임워크가 이미 처리하는 항목(예: Prisma의 SQL 파라미터 바인딩)은 "해당 없음"으로 분류한다.
- Rate Limiter는 MVP에서 인메모리로 충분하지만, 프로덕션 스케일에서는 Redis 필요를 안내한다.
- Audit Logging은 MVP에서 console.log JSON으로 시작하되, 프로덕션에서 외부 서비스 연동을 안내한다.
